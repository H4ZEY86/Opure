## 1. Decision Summary

Opure should adopt a **layered, risk-driven and evidence-based testing strategy**.

The primary coded-test stack should use:

- a stable **xUnit.net v3** release as the default .NET test framework;
- **Microsoft Testing Platform** as the preferred test runner where the pinned .NET SDK and development environments support it;
- temporary VSTest compatibility only while required by supported tools;
- built-in xUnit assertions as the default assertion library;
- handwritten fakes and deterministic test doubles as the default substitution strategy;
- `Avalonia.Headless.XUnit` for fast control, binding, layout and input tests;
- real-window Appium tests for a small set of critical Windows user journeys and accessibility checks;
- `coverlet.MTP` or another approved MTP-compatible coverage extension for managed-code coverage;
- BenchmarkDotNet for isolated microbenchmarks where statistical benchmarking is appropriate;
- custom Opure process, fault-injection, recovery, security and architecture-test harnesses;
- and explicit release evidence generated by the same commands used in continuous integration.

Testing must be organised into distinct layers:

1. compiler and static-analysis checks;
2. unit tests;
3. contract and compatibility tests;
4. architecture tests;
5. component tests;
6. headless desktop tests;
7. integration tests;
8. real-process topology tests;
9. persistence and migration tests;
10. security and adversarial tests;
11. fuzz and property tests;
12. fault-injection and recovery tests;
13. performance and resource tests;
14. real-window end-to-end tests;
15. accessibility tests;
16. packaging, installation and update tests;
17. and release acceptance tests.

Opure must not treat:

- code coverage;
- test count;
- snapshot count;
- passing AI-generated tests;
- or a green continuous-integration run

as sufficient proof that the intended product behaviour has been delivered.

Tests must verify the Charter, specifications, ADRs, system boundaries, recovery behaviour and developer-visible outcomes.

## 3. Context

Opure is a local software engineering platform containing:

- a desktop application;
- a trusted Runtime;
- in-process services;
- trusted workers;
- isolated Plugin Hosts;
- local IPC;
- SQLite persistence;
- a Secrets Vault;
- AI provider adapters;
- MCP integrations;
- build and test execution;
- Git operations;
- patch transactions;
- workflows;
- approvals;
- recovery;
- and developer-facing trust records.

The platform handles private source code and state-changing engineering operations.

A defect can cause:

- data loss;
- secret disclosure;
- incorrect patch application;
- hidden network transmission;
- repository corruption;
- workflow duplication;
- false success;
- inaccessible UI;
- uncontrolled resource consumption;
- or loss of developer confidence.

The test strategy must therefore cover more than ordinary unit correctness.

It must provide evidence that:

- boundaries are enforced;
- processes fail independently;
- commands are cancellable;
- transactions recover;
- secrets do not leak;
- providers remain replaceable;
- project isolation holds;
- the UI remains accessible;
- and release artefacts install and recover safely.

Opure will also use AI to propose code.

AI-generated code and tests can reinforce the same mistaken interpretation.

Independent acceptance evidence is therefore essential.

## 4. Problem Statement

Opure requires a practical testing architecture that gives a small team fast feedback during development while providing deep evidence for security, recovery, compatibility, performance, accessibility and release safety across a multi-process local platform.

## 8. Constraints

Known constraints include:

- Windows 11 is the first supported platform.
- C#/.NET is the proposed implementation stack.
- Avalonia is the proposed desktop framework.
- xUnit and MTP versions must be pinned.
- Some real Windows tests require an interactive desktop session.
- Appium desktop automation is slower and more fragile than headless tests.
- SQLite crash tests require real files and processes.
- Process supervision tests require real child processes.
- Secret tests must use synthetic secrets.
- Performance tests require controlled machines.
- Local AI tests must not require one specific model for every developer.
- Cloud-provider tests must not run by default.
- Tests must work offline unless explicitly classified as external.
- CI environments may have different timing from developer hardware.
- Same-user security boundaries cannot be fully simulated by unit tests.
- Test artefacts may contain private data and must be treated accordingly.
- The suite must remain maintainable by a small team.
- Test duration must remain bounded.
- A flaky release-blocking test cannot be ignored indefinitely.

- Model-quality tests can use recorded or deterministic fixtures without asserting exact prose.

- Release tests can run on clean Windows virtual machines.
- Manual exploratory testing remains valuable and is not replaced completely.

# 12. Option A — xUnit.net v3

## 12.1 Description

Use a stable xUnit.net v3 release for the majority of C# coded tests.

Use Microsoft Testing Platform as the preferred execution platform.

---

## 12.2 Advantages

- Strong .NET ecosystem adoption.
- Modern async-test support.
- Stand-alone test executables.
- Native Microsoft Testing Platform integration.
- Good command-line execution.
- Good IDE support.
- Good theory and data-test model.
- Good fixture support.
- Conservative parallelism defaults.
- Avalonia headless xUnit integration.
- Familiar to many contributors.
- Small assertion surface.
- Strong fit for test-project isolation.
- Supports custom traits and test-case order controls where justified.
- Supports Native AOT testing scenarios.
- Open-source framework.
- Good support for deterministic test class construction.

---

## 12.3 Disadvantages

- Parallelism requires careful governance.
- Fixture lifetime can surprise new contributors.
- Test-output capture differs from some frameworks.
- V3 and future v4 package transitions require version discipline.
- Some third-party tooling may lag MTP support.
- Assembly-level configuration can be fragmented.
- xUnit's conventions may encourage constructor setup even when explicit setup improves clarity.
- UI and process tests still need custom harnesses.
- Exact runner behaviour differs between native xUnit, VSTest and MTP.

---

## 12.4 Risks

- Mixing xUnit v2 and v3 packages.
- Accidentally adopting pre-release xUnit packages.
- Running different test platforms locally and in CI.
- Incorrect parallel execution of shared resources.
- Traits becoming inconsistent.
- Fixture state leaking.
- MTP command-line differences causing false assumptions.
- IDE runner incompatibility.
- Tests depending on execution order.

---

## 12.5 Mitigations

- central package management;
- stable package pinning;
- one runner configuration;
- explicit collections for shared resources;
- no test-order dependency;
- deterministic fixtures;
- architecture tests for package references;
- and a standard test command.

---

## 12.6 Estimated Adoption Cost

- **Initial implementation:** Low to Moderate
- **Operational complexity:** Low to Moderate
- **Migration difficulty:** Low
- **Replacement difficulty:** Moderate

# 17. Test Platform Decision

Microsoft Testing Platform is preferred because it is the modern .NET testing platform and supports stand-alone test modules, extensions and current `dotnet test` integration.

---

## 17.1 MTP Primary

The pinned SDK should configure:

```json
{
  "test": {
    "runner": "Microsoft.Testing.Platform"
  }
}
```

when supported by the final selected SDK and package versions.

---

## 17.2 VSTest Compatibility

During the transition period, test projects may retain:

- `xunit.runner.visualstudio`;
- and `Microsoft.NET.Test.Sdk`

only while required by supported IDEs or CI environments.

The compatibility path must not create two different semantic test suites.

---

## 17.3 Native Test Executables

xUnit v3 test projects should remain runnable directly as executables where useful.

CI may use:

- `dotnet test`;
- or built test executables

through one centrally governed wrapper.

# 18. Decision

Opure will provisionally use:

> **A stable xUnit.net v3 release with Microsoft Testing Platform as the primary test runner, layered test projects, Avalonia headless xUnit tests, a small real-window Appium suite, MTP-compatible managed coverage and dedicated Opure security, process, recovery and performance harnesses.**

This decision does not approve:

- a single giant test project;
- all tests running in parallel;
- mocking every dependency;
- snapshot tests as primary correctness evidence;
- exact AI prose assertions;
- automatic retry hiding flaky tests;
- production secrets in tests;
- cloud tests in the default local suite;
- or code coverage as a release-quality score.

This decision is:

- [ ] Permanent until superseded
- [x] Provisional pending prototype evidence
- [ ] Experimental only
- [ ] Limited to unit tests
- [ ] Limited to Windows permanently

# 19. Rationale

xUnit.net v3 provides the strongest overall fit because it offers:

- a modern test model;
- MTP integration;
- direct executable test modules;
- broad .NET familiarity;
- and Avalonia headless support.

Microsoft Testing Platform provides a modern runner and extension architecture.

Headless Avalonia tests provide fast UI-component feedback.

Appium real-window tests cover behaviours that headless tests cannot prove, especially:

- native windowing;
- focus;
- platform integration;
- and accessibility-tree behaviour.

Custom harnesses remain necessary because no generic unit-test framework alone can prove:

- process-tree cancellation;
- Runtime recovery;
- plugin isolation;
- secret non-disclosure;
- database crash safety;
- or installer rollback.

---

# 20. Test Portfolio

Opure should use a portfolio rather than a simplistic pyramid.

The intended shape is:

- many fast deterministic unit and contract tests;
- many architecture and component tests;
- a moderate integration and process suite;
- targeted security and recovery suites;
- targeted performance suites;
- and a small critical end-to-end UI suite.

---

## 20.1 Why Not One Pyramid

Opure includes security and recovery boundaries that are not adequately represented by a conventional unit-versus-end-to-end pyramid.

Security, recovery and architecture tests cut across layers.

---

# 21. Test Levels

## 21.1 Level 0 — Build and Static Verification

Includes:

- compiler;
- nullable analysis;
- warnings as errors;
- analyzers;
- formatting;
- package lock verification;
- dependency vulnerability checks;
- generated-code consistency;
- and schema compilation.

---

## 21.2 Level 1 — Unit Tests

Test one class, function or small aggregate in memory with deterministic collaborators.

---

## 21.3 Level 2 — Contract Tests

Verify:

- service contracts;
- Protocol Buffer compatibility;
- error codes;
- event envelopes;
- version negotiation;
- and provider-adapter behaviour.

---

## 21.4 Level 3 — Architecture Tests

Verify dependency and ownership rules.

---

## 21.5 Level 4 — Component Tests

Test one service or component with its real internal implementation and controlled external dependencies.

---

## 21.6 Level 5 — Headless Desktop Tests

Test Avalonia controls, binding, layout and input without a visible OS window.

---

## 21.7 Level 6 — Integration Tests

Test multiple real services, SQLite stores, IPC components or adapters together.

---

## 21.8 Level 7 — Real-Process Tests

Launch actual Desktop, Runtime, worker, Plugin Host or helper executables.

---

## 21.9 Level 8 — Security and Adversarial Tests

Test denial, isolation, tamper, leakage and malicious input.

---

## 21.10 Level 9 — Recovery and Fault-Injection Tests

Interrupt operations and prove exact recovery.

---

## 21.11 Level 10 — Performance and Resource Tests

Measure latency, throughput, memory, CPU, disk and cancellation.

---

## 21.12 Level 11 — Real-Window End-to-End Tests

Drive the compiled application through the platform accessibility tree.

---

## 21.13 Level 12 — Packaging and Release Tests

Verify installation, upgrade, rollback, uninstall and recovery on clean systems.

# 23. Test Naming

Test names should state:

- operation;
- condition;
- expected observable result.

Preferred form:

```text
Method_WhenCondition_ShouldOutcome
```

or a readable equivalent.

Avoid names that merely restate implementation details.

---

# 24. Arrange, Act and Assert

Tests should make:

- initial state;
- action;
- and expected outcome

clear.

Comments are optional when structure is evident.

---

# 25. One Behaviour Per Test

A test should normally prove one coherent behaviour.

Multiple assertions are acceptable when they describe one outcome.

---

# 26. Assertions

Built-in xUnit assertions are the default.

---

## 26.1 Why Built-In First

Benefits:

- minimal dependency surface;
- predictable licence;
- contributor familiarity;
- and no hidden global assertion configuration.

---

## 26.2 Custom Assertions

Opure may provide focused assertions for domain concepts such as:

- service result;
- error envelope;
- patch state;
- Trust record;
- secret redaction;
- process exit;
- and recovery outcome.

---

## 26.3 Assertion Messages

Assertions should reveal:

- expected state;
- actual state;
- and relevant safe evidence.

They must not print secret values.

---

# 27. Test Doubles

Preferred order:

1. real value object;
2. deterministic in-memory implementation;
3. handwritten fake;
4. controlled stub;
5. spy;
6. mocking framework only when justified.

---

## 27.1 Handwritten Fakes

Fakes should expose meaningful test state and remain reusable.

Examples:

- fake clock;
- fake random source;
- fake provider server;
- fake Network Gateway;
- fake approval service;
- fake filesystem;
- fake process launcher;
- fake Trust recorder;
- and fake Vault broker.

---

## 27.2 Mocking Framework

No mocking framework is selected as a foundation dependency.

A later decision may approve one if handwritten fakes become disproportionately costly.

---

## 27.3 Avoid Interaction Worship

Tests should prefer observable outcome over verifying every internal call.

Interaction assertions are appropriate for:

- security boundary;
- no network call;
- no file write;
- no secret retrieval;
- and idempotency.

---

# 28. Determinism

Tests should control:

- time;
- random values;
- identifiers;
- environment;
- current directory;
- locale;
- time zone;
- process environment;
- network;
- filesystem roots;
- provider responses;
- and scheduling where practical.

---

## 28.1 Time

Use an injected time provider.

Tests must not rely on wall-clock sleeping for ordinary state transitions.

---

## 28.2 Randomness

Use injected randomness or recorded seeds.

Security tests may use real cryptographic randomness but should record safe reproduction metadata where possible.

---

## 28.3 Identifiers

Use deterministic factories in tests.

Do not assert against unpredictable production IDs unless testing their format or uniqueness.

---

## 28.4 Culture

Tests involving parsing or formatting should set culture explicitly.

---

## 28.5 Time Zone

Store and compare UTC.

Display tests set the target time zone explicitly.

---

# 29. Test Isolation

Every test should receive isolated:

- temporary directory;
- Runtime profile;
- database files;
- IPC namespace;
- Vault;
- diagnostic directory;
- ports where required;
- and process identity.

---

## 29.1 Production Isolation

Tests must never use:

- the developer's normal Opure profile;
- real Vault files;
- real project state;
- or active Runtime endpoint.

---

## 29.2 Endpoint Randomisation

Every process test should use a random endpoint namespace.

---

## 29.3 Cleanup

Cleanup should be best effort after success and failure.

Failed tests may preserve artefacts through an explicit retained-fixture policy.

---

# 30. Parallelism

Parallelism is allowed only when resource isolation is proven.

---

## 30.1 Safe Parallel Tests

Pure unit tests may run in parallel.

---

## 30.2 Serial Collections

Tests sharing:

- UI application instance;
- global environment;
- Windows registry;
- named operating-system resource;
- process supervisor;
- performance machine;
- or installer

must use controlled serial collections.

---

## 30.3 No Order Dependency

Tests must pass in:

- full suite;
- isolated execution;
- random order;
- and repeated execution.

---

# 31. Unit Tests

Unit tests should cover:

- value objects;
- state machines;
- policy decisions;
- error translation;
- path classification;
- contract validation;
- workflow stage logic;
- patch planning;
- risk calculation;
- redaction;
- retention;
- and cancellation-state transitions.

---

## 31.1 Unit Test Speed

A typical unit test should complete in milliseconds.

The unit suite should remain suitable for continuous local execution.

---

## 31.2 No Filesystem by Default

A test using a real filesystem is an integration or component test unless the filesystem behaviour itself is the unit under test.

---

# 32. Contract Tests

Contract tests are required for every process and service boundary.

---

## 32.1 Protocol Buffer Tests

Verify:

- field-number preservation;
- removed-field reservation;
- unknown-field behaviour;
- unknown enum values;
- optional-field defaults;
- message-size limits;
- and version negotiation.

---

## 32.2 Golden Contract Fixtures

Maintain binary and JSON diagnostic fixtures for supported protocol versions.

Golden fixtures are reviewed assets, not snapshots updated blindly.

---

## 32.3 Old and New Compatibility

Test:

- previous Desktop with current Runtime;
- current Desktop with previous supported Runtime;
- previous worker with current Runtime where supported;
- current Plugin Host with supported plugin SDK versions;
- and incompatible versions failing clearly.

---

## 32.4 Error Contract

Verify stable Opure error codes across transport mappings.

---

# 33. Architecture Tests

Architecture tests are release-blocking.

They should verify:

- project-reference direction;
- no circular dependencies;
- no Avalonia dependency in domain services;
- no provider SDK outside adapters;
- no direct protected file write outside Patch Service;
- no direct secret access outside Vault;
- no direct MCP access outside MCP Gateway;
- no direct external network access outside approved adapters;
- no Desktop database access;
- no Plugin Host trusted-core reference;
- no worker ownership of authoritative stores;
- no unsupervised process creation;
- and no cross-service persistence reference.

---

## 33.1 Implementation

Initial architecture tests should use:

- assembly reflection;
- project graph inspection;
- generated dependency manifests;
- and Roslyn analysis where syntax or symbol inspection is required.

A third-party architecture-testing library is not required initially.

---

## 33.2 Source Rules

Some rules require source analysis.

Examples:

- forbidden API use;
- secret type serialisation;
- direct `HttpClient` construction;
- direct file write;
- command-line secret field;
- and unsafe code.

---

# 34. Component Tests

- parallelism comparison;
- coverage comparison;
- and CI validation.

The build should fail if expected test assemblies disappear unexpectedly.

- test assembly;
- category;
- count;
- skipped;
- quarantined;

Count is diagnostic, not quality.

A skipped test requires:

- reason;
- issue;
- owner;
- and expiry

unless it is platform-inapplicable by design.

A Windows-only test skipped on Linux is not flaky.

Show:

- test;
- owner;
- age;
- last failure;
- and expiry.

Release evidence should be retained according to release policy.

It may contain sensitive synthetic artefacts but not production data.

Ordinary temporary test data should be deleted.

Failed-test fixtures may be retained for a bounded period.

# 130. Developer Experience

A contributor should be able to:

1. restore;
2. build;
3. run the fast suite;
4. understand failure;
5. and reproduce CI

without proprietary cloud access.

The fast suite must pass with the network disabled.

Fake external servers should support:

- latency;
- disconnect;

- partial response;
- rate limit;
- authentication failure;
- and retry-after.

# 133. Cancellation Tests

Every long-running service should test:

- cancellation before admission;
- cancellation during work;
- cancellation after external side effect;
- forced termination;
- and final state.

---

# 134. Timeout Tests

Timeout tests should use deterministic or accelerated time where practical.

---

# 135. Backpressure Tests

Test:

- slow Desktop;
- output flood;
- token flood;
- event flood;
- log queue full;
- and IPC stream full.

---

# 136. Idempotency Tests

Test duplicate:

- command;
- event;
- outbox delivery;
- workflow recovery;
- patch request;
- and secret lease request.

---

# 137. Concurrency Tests

Test:

- two commands for one aggregate;
- project close during workflow;
- Runtime shutdown during patch;
- secret revoke during provider request;
- and backup during writes.

---

# 138. Race Detection

Use repeated stress and controlled schedulers to expose races.

A specific race-detection tool is not selected.

---

# 139. Deadlock Tests

Tests should impose deadlines and capture process dumps for hangs in isolated test environments.

---

# 140. Crash and Hang Diagnostics

MTP extensions may be used for crash and hang evidence after package review.

They must respect secret and privacy rules.

---

# 141. Test Telemetry

Test-platform telemetry must be reviewed and disabled where required by project policy.

The repository should not silently transmit test usage.

# 142. Security of Test Reports

Reports may reveal:

- machine;
- paths;
- project structure;
- stack traces;
- and commands.

CI publication should apply access controls and retention.

# 143. Path Redaction

Public test artefacts should redact local user paths.

# 144. Flaky Environment

A machine failure, antivirus interference or CI outage should be distinguished from a product-test failure.

Before expensive suites, validate:

- disk space;
- SDK;
- permissions;
- interactive session;
- GPU where required;
- and test dependencies.

- record existence;

- structured logging;
- trace correlation;
- metric names;

- provider;

- failure;
- and recovery

without appearing in prohibited stores.

Critical workflows should confirm that a developer can still:

Safe Mode should start with:

- plugins disabled;
- MCP restricted;
- optional providers disabled;
- core diagnostics;
- and recovery capabilities.

# 155. Recovery Mode Tests

Recovery Mode should work when:

- ordinary Runtime startup fails;
- database migration fails;
- Vault is unavailable;

# 156. Installer Security Tests

Test:

- signature validation;
- path permissions;
- no machine-wide service by default;
- per-user install;

Uninstall should distinguish:

- projects;
- Vault;
- backups;
- diagnostics;

Maintain supported combinations for:

A failed upgrade must restore:

- binaries;
- compatible databases;
- keyring;
- and launchability

# 162. Test Review Checklist

Reviewers should ask:

- Is the test at the lowest effective layer?
- Is the oracle independent?
- Is it deterministic?
- Does it prove failure behaviour?
- Does it leak data?
- Does it overfit implementation?
- Can it reproduce?
- Does it need real process or UI?
- Is the maintenance cost justified?

# 168. Alternatives Rejected

## 168.1 NUnit as the Primary Framework

Not selected because xUnit.net v3 provides an equally capable modern .NET model, strong MTP integration and a natural fit with the proposed Avalonia headless path.

---

## 168.2 MSTest as the Primary Framework

Not selected because it provides no decisive benefit over xUnit.net v3 for Opure's architecture.

---

## 168.3 TUnit as the Primary Framework

Not selected because its younger ecosystem adds avoidable foundation risk.

---

## 168.4 VSTest as the Permanent Primary Platform

Not selected because Microsoft Testing Platform is the modern direction for current .NET test execution.

VSTest remains a compatibility bridge only.

---

## 168.5 End-to-End Tests as the Main Strategy

Rejected because they are slower, harder to diagnose and insufficient for combinatorial security and recovery logic.

---

## 168.6 Unit Tests Only

Rejected because unit tests cannot prove process, filesystem, IPC, accessibility, installer or crash behaviour.

---

## 168.7 Coverage as the Quality Gate

Rejected because execution coverage does not prove correct assertions, correct requirements or safe recovery.

---

## 168.8 Snapshot-Heavy Testing

Rejected as a general strategy because broad snapshot updates can approve unintended changes without understanding.

Snapshots remain appropriate for selected reviewed artefacts.

---

## 168.9 Automatic Retry as Flake Management

Rejected because retry can hide nondeterminism and false failure.

Retries remain diagnostic and must preserve the first result.

# 174. Final Decision Statement

> **Opure will provisionally use a layered, risk-driven testing strategy built on a stable xUnit.net v3 release and Microsoft Testing Platform, combining fast deterministic unit, contract, architecture and headless desktop tests with real-process, security, recovery, performance, accessibility and release suites, because trustworthy software-engineering automation requires independent evidence of behaviour, isolation and recoverability rather than test counts, coverage percentages or AI-generated confidence.**
