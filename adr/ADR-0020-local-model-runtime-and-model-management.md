## 1. Decision Summary

Opure should implement local model support through a **provider-neutral Local Model Runtime boundary**, with a pinned and Opure-built `llama.cpp` runtime as the first production adapter and verified GGUF model artefacts as the first production model format.

The initial local model architecture should:

* keep the AI Router authoritative over all local inference requests;
* represent local runtimes and local models through immutable, revisioned records;
* separate:

  * model source;
  * model artefact;
  * model identity;
  * model runtime;
  * execution profile;
  * loaded instance;
  * and inference session;
* run production local inference outside `Opure.Runtime` and `Opure.Desktop`;
* launch the first runtime as a supervised `llama-server` process;
* build the runtime from a pinned upstream `llama.cpp` commit in Opure's release pipeline;
* never install `llama.cpp` dynamically through Winget, Conda, a package manager or an unpinned release during normal product use;
* package and verify every runtime binary, native dependency, licence and SBOM;
* support a tested NVIDIA CUDA backend on the Windows 11 reference machine;
* support a tested CPU backend as a compatibility and recovery path;
* permit Vulkan only through a separately tested optional execution profile;
* prohibit automatic backend changes when they alter performance, precision, memory use or behaviour;
* prohibit automatic fallback from GPU to CPU during an active request;
* permit the user or scheduler to select a different approved execution profile before retry;
* use GGUF as the first accepted production weight container;
* support text-generation, code-generation, fill-in-the-middle and embedding model roles where the runtime and model metadata prove compatibility;
* keep multimodal local inference behind a separate tested model-role profile;
* defer direct Safetensors, PyTorch pickle, ONNX and vendor-engine model loading in Version 1;
* prohibit loading pickle-based model artefacts;
* recognise Safetensors as safer than pickle but treat it as a future conversion or runtime-adapter input rather than the first production inference format;
* defer ONNX Runtime GenAI as a production adapter while its generate API remains documented as preview;
* allow future runtime adapters without changing AI Router contracts;
* store model files in an Opure-owned content-addressed Model Store;
* store every verified file by SHA-256;
* keep model artefacts immutable after verification;
* keep mutable runtime caches, KV caches, logs, temporary files and benchmark results outside the immutable artefact;
* never edit or convert a source model in place;
* never trust a filename, repository name, model card or display name as model identity;
* bind model identity to:

  * source;
  * exact source revision;
  * exact file list;
  * exact file hashes;
  * format metadata;
  * architecture;
  * quantisation;
  * licence evidence;
  * and importer version;
* require an explicit Model Manifest generated by trusted Opure code;
* require a Model Licence Record before activation;
* distinguish:

  * licence identifier;
  * licence text;
  * acceptable-use terms;
  * gated-access terms;
  * commercial-use restrictions;
  * attribution requirements;
  * redistribution rights;
  * and unknown conditions;
* never describe a model as “open source” solely because weights can be downloaded;
* never redistribute a model unless its licence and source terms clearly permit redistribution;
* permit user-managed local imports when redistribution is not permitted;
* require explicit acknowledgement for gated models and custom licences;
* use a browser or source-specific authenticated flow for access acceptance where required;
* store source access tokens only in the Secrets Vault;
* never place source tokens in model manifests, paths, logs, command lines or runtime environments;
* download models only through a trusted Model Acquisition Service and the Network Gateway;
* perform no model download during inference;
* perform no automatic update check from the model runtime;
* perform no automatic runtime download;
* perform no automatic model conversion;
* perform no automatic model quantisation;
* perform no automatic remote-code execution;
* pin Hugging Face or other repository downloads to a full immutable revision identifier where available;
* reject mutable `main`, `latest` or branch-only references for production installation unless they are resolved and recorded to a concrete commit before download;
* allow only an explicit file allowlist;
* reject unexpected executable files, scripts, pickles, shared libraries and nested archives;
* capture model card, licence, configuration, tokenizer metadata and required auxiliary files as evidence;
* treat repository malware or pickle scanning badges as useful external evidence but not as a substitute for Opure verification;
* independently scan every downloaded file;
* compute hashes while downloading;
* quarantine downloads before import;
* validate redirects and all content hosts;
* enforce byte, file-count and decompression limits;
* support resumable downloads only when the source proves byte-range identity and the final full-file hash is verified;
* verify free disk space before acquisition;
* produce an exact download preview showing source, revision, files, sizes, licence, gated terms, expected runtime and storage impact;
* require explicit developer approval before network download;
* support explicit offline import from local files or removable media;
* repeat all format, hash, licence and malware checks for offline imports;
* prohibit model-code execution during import;
* prohibit Python `trust_remote_code`;
* prohibit model repository scripts;
* prohibit arbitrary conversion scripts from downloaded repositories;
* use only Opure-owned, version-pinned conversion tools;
* run conversion and quantisation in isolated workers;
* treat converted artefacts as new derived models with complete provenance;
* retain the original source artefact only according to policy and licence;
* never claim converted output is identical to the source;
* record quantisation method, parameters, tool commit and calibration data where applicable;
* record model architecture and runtime compatibility from trusted parsing, not only model-card text;
* reject unsupported or inconsistent GGUF metadata;
* validate GGUF structure before runtime loading;
* cap tensor counts, dimensions, metadata entries, string lengths, alignment and file offsets;
* reject overlapping, out-of-range or malformed tensor data;
* never map an unverified model into a production runtime;
* launch `llama-server` with:

  * exact verified executable;
  * exact verified model path;
  * fixed argument vector;
  * loopback-only binding;
  * a random high port selected by Opure;
  * a random per-process API key;
  * Web UI disabled;
  * built-in agent features disabled;
  * CORS restricted or disabled;
  * slot-management endpoints disabled where not required;
  * property-mutation endpoints disabled;
  * metrics disabled unless a trusted local diagnostic session enables them;
  * prompt-cache persistence disabled;
  * model download flags prohibited;
  * remote RPC backends prohibited;
  * and a minimal environment;
* treat `llama-server`'s OpenAI-compatible API as a runtime transport only;
* not infer OpenAI API semantics, privacy or feature support from that compatibility label;
* use a private AI Router adapter with an allowlisted endpoint subset;
* accept no user-supplied runtime URL for an Opure-managed process;
* verify the child process, port owner, API key and Runtime instance before inference;
* use a dedicated per-instance random API prefix where supported;
* reject browser, plugin and unrelated-process access;
* use Windows Firewall or process isolation controls where practical;
* place every managed runtime in a non-breakaway Job Object;
* enable kill-on-job-close;
* restrict child-process creation;
* enforce process-count, memory, CPU and lifetime limits;
* pass no unnecessary inheritable handles;
* set safe DLL search behaviour;
* run from an Opure-owned working directory;
* pass no provider credentials or unrelated environment variables;
* deny runtime outbound network where practical;
* record any platform limitation that prevents complete network denial;
* require a future stronger local IPC or embedded-runtime adapter before claiming that loopback HTTP is a hardened security boundary;
* use loopback API authentication and process supervision as defence in depth;
* stop the runtime after a configurable idle period;
* unload models by terminating the process;
* treat process termination as the authoritative release of GPU and native resources;
* never rely on a native library unloading safely inside the trusted Runtime;
* run one model per process initially;
* permit only one loaded generation model by default on the 16 GiB reference GPU;
* permit a small embedding model concurrently only after measured evidence;
* make multi-model concurrency explicit and resource-budgeted;
* prohibit unbounded model multiplexing;
* schedule model loads through a Local Model Scheduler;
* reserve memory for Desktop, Runtime, build tools and the operating system;
* estimate:

  * model weights;
  * runtime buffers;
  * KV cache;
  * batch buffers;
  * multimodal projector;
  * and safety margin;
* query actual system RAM and GPU memory before load;
* deny a load that cannot meet the selected performance profile safely;
* support CPU+GPU hybrid offload only as an explicit execution profile;
* not silently increase system RAM or pagefile pressure;
* avoid `mlock` by default on Windows;
* use memory mapping only after validating model-store filesystem and platform behaviour;
* keep model files on a local fixed disk by default;
* reject or warn for network shares, cloud-synchronised folders, removable media and unsupported filesystems;
* copy user-imported model files into the Model Store by default;
* permit reference-in-place only as an advanced, visibly fragile mode;
* revalidate referenced files before every load;
* treat any changed file hash as a new or corrupted artefact;
* expose Eco, Balanced, Performance and Turbo execution modes;
* make Balanced the initial default;
* keep security limits unchanged across performance modes;
* let modes adjust:

  * context length;
  * batch size;
  * thread count;
  * GPU offload;
  * concurrency;
  * idle timeout;
  * and memory reserve;
* never let Turbo override thermal, memory or policy hard limits;
* support per-model benchmark and compatibility evidence;
* separate model quality evaluation from runtime compatibility;
* never automatically select a model solely from synthetic benchmark scores;
* display expected memory, context and performance characteristics before activation;
* preserve exact runtime and model provenance in every local request receipt;
* record:

  * runtime package revision;
  * runtime executable hash;
  * backend;
  * model manifest revision;
  * model file hashes;
  * quantisation;
  * context configuration;
  * seed where used;
  * sampling parameters;
  * output limits;
  * and actual usage;
* keep prompt and output content out of routine diagnostics;
* expose local model health, loading, memory, throughput and failure state through the trusted Gateway;
* never expose the runtime's raw diagnostic endpoints directly to Desktop;
* treat local model output as untrusted proposed content;
* use normal Patch, Tool and Workflow review boundaries;
* support explicit model disablement, quarantine, deletion and data cleanup;
* stop all active instances before deleting model files;
* preserve active request receipts and provenance after model deletion;
* provide a deletion preview showing immutable artefacts, mutable cache, derived models and references;
* not promise forensic secure deletion;
* support model update as installation of a new immutable revision;
* never replace files in an installed model revision;
* never reuse one model version or source revision for different bytes;
* require review when model licence, architecture, quantisation, chat template, tokenizer, context limit or source revision changes;
* not migrate workflows to a new model automatically;
* support rollback by reselecting a retained verified model revision;
* not roll back a model if the old artefact is revoked, corrupted or licence-incompatible;
* support quarantine on:

  * hash mismatch;
  * malformed format;
  * licence conflict;
  * malware finding;
  * model-source compromise;
  * runtime crash loop;
  * unexpected network activity;
  * or model identity mismatch;
* preserve evidence while preventing new loads;
* keep Stable, Preview and Development model registries and execution policies separate;
* permit physical deduplication only if authority, licences and references remain independently represented;
* and defer:

  * public model marketplace;
  * automatic model recommendations;
  * automatic quantisation;
  * automatic conversion;
  * remote model execution workers;
  * distributed inference;
  * multi-GPU inference;
  * speculative decoding;
  * LoRA activation;
  * model merging;
  * fine-tuning;
  * provider-side model downloads;
  * and cross-platform local runtimes.

The first runtime decision should be:

> **Use an Opure-built and pinned `llama.cpp` `llama-server` process with the GGUF format, a tested CUDA execution profile for the NVIDIA reference workstation, a CPU recovery profile, loopback-only authenticated transport, no runtime network or model download authority, one model per supervised process and immutable content-addressed model storage.**

The first runtime is not the permanent universal runtime.

Opure should preserve an internal adapter boundary for future:

* ONNX Runtime GenAI;
* Windows ML;
* DirectML;
* Vulkan;
* vendor-optimised runtimes;
* embedding-specific runtimes;
* multimodal runtimes;
* and future cross-platform backends.

The selected trust chain is:

```text
Explicit model source
    ↓
Exact immutable source revision
    ↓
Licence and access review
    ↓
Network Gateway download
    ↓
Quarantine and independent scanning
    ↓
Full-file hashes and format validation
    ↓
Model Manifest and provenance
    ↓
Immutable content-addressed Model Store
    ↓
Approved Runtime Package and Execution Profile
    ↓
Resource preflight
    ↓
Supervised no-network local process
    ↓
Authenticated loopback inference
    ↓
AI Router validation
    ↓
Untrusted model result
    ↓
Reviewable Opure action
    ↓
Trust Centre receipt
```

## 3. Context

ADR-0019 establishes local inference as Opure's preferred provider path.

A useful local-first product still needs to answer:

* which runtime executes models;
* which model formats are accepted;
* how model files are obtained;
* how licences and gated terms are handled;
* how files are verified;
* where models are stored;
* how runtime binaries are trusted;
* how GPU and RAM are allocated;
* how context length affects memory;
* how models are loaded and unloaded;
* how updates are represented;
* how conversion is controlled;
* how derived models retain provenance;
* how local endpoints are authenticated;
* how runtime network access is prevented;
* how crashes are contained;
* how the user understands expected performance;
* and how model deletion and recovery work.

Local does not automatically mean safe.

A local model package may contain:

* malformed tensor data;
* malicious pickle objects;
* native libraries;
* scripts;
* custom code;
* ambiguous licences;
* model-card instructions;
* a tokenizer;
* chat templates;
* multimodal projectors;
* or adapters.

A local runtime is native executable code.

It may:

* open network connections;
* read files;
* write prompt logs;
* create caches;
* spawn child processes;
* consume all GPU memory;
* consume all system RAM;
* become unresponsive;
* or expose an unauthenticated local HTTP service.

A model repository may change after the user selects it.

A branch name is mutable.

A model filename can be replaced.

A model card is publisher-supplied metadata.

A repository scan badge is not a local proof.

A quantised conversion may be produced by an unknown tool.

A model licence may permit use but prohibit redistribution.

A gated model may require personal information and browser acceptance.

A model may technically load but have an incompatible chat template or tokenizer.

A model may fit in VRAM at one context length and fail at another.

A 16 GiB GPU cannot safely load arbitrary models and contexts without planning.

Opure therefore needs a complete model-lifecycle architecture, not only a local API URL.

## 4. Problem Statement

Opure requires a local model runtime and model-management architecture that provides useful GPU-accelerated inference on Windows 11 while preserving model provenance, licence visibility, immutable storage, safe acquisition, strict format validation, bounded resource use, runtime process containment, offline operation, predictable updates and provider-neutral future evolution.

## 8. Constraints

Known constraints include:

* The first target is Windows 11 x64.
* The reference workstation has an NVIDIA RTX 5070 Ti with 16 GiB VRAM.
* The reference CPU is a Ryzen 9 5950X.
* The reference machine has 32 GiB system RAM.
* `llama.cpp` is a rapidly evolving native project.
* Its command-line and server APIs can change.
* It supports GGUF and multiple hardware backends.
* It can run an OpenAI-compatible server.
* It can use CUDA on NVIDIA hardware.
* It supports CPU and GPU hybrid inference.
* It supports many quantisation types.
* Its server can expose authentication, health, metrics, slots, embeddings, reranking, Web UI and agent features.
* Some server endpoints are enabled by default and must be explicitly restricted.
* GGUF is extensible and self-describing but still requires defensive parsing.
* Hugging Face model repositories are version-controlled and can be pinned to full commits.
* Hub downloads may redirect to multiple storage and CDN hosts.
* gated access requires an authenticated user token and browser acceptance;
* Hub malware and pickle scanning is useful but best effort;
* pickle model files can execute arbitrary code when loaded;
* Safetensors is designed as a non-pickle tensor format;
* ONNX Runtime GenAI currently documents its high-level generate API as preview;
* model licences range from permissive to custom, gated, research-only or unknown;
* model artefacts can be tens of gigabytes;
* local model processes can consume substantial VRAM, RAM, CPU, disk and power;
* and Opure must remain responsive while local inference runs.

## 9. Assumptions

This decision assumes:

* a pinned `llama.cpp` build can run reliably on the reference GPU and CPU;
* Opure can package required native dependencies lawfully;
* `llama-server` can bind to loopback and require an API key;
* Web UI and unnecessary endpoints can be disabled;
* Network Gateway and Windows controls can prevent or detect unintended runtime network access;
* GGUF validation can occur before launch;
* models can be represented by immutable manifests;
* Hugging Face downloads can be resolved to full commit revisions;
* local source tokens can remain in the Vault;
* a content-addressed Model Store is practical on NTFS;
* Process Supervisor can enforce Job Object limits;
* the scheduler can estimate resource demand conservatively;
* model load failure can fail closed without cloud fallback;
* process termination reliably releases native runtime resources;
* and future adapters can implement the same Local Model Runtime contracts.

## 10. Current Technical Evidence

Official and primary project documentation available on 18 July 2026 establishes that:

* `llama.cpp` is a C/C++ local inference project supporting multiple quantisation levels;
* it supports custom CUDA kernels for NVIDIA GPUs;
* it supports Vulkan and other backends;
* it supports CPU and GPU hybrid inference;
* its primary model format is GGUF;
* `llama-server` can bind to `127.0.0.1`;
* `llama-server` supports API-key authentication;
* it exposes health and OpenAI-compatible endpoints;
* it can expose optional metrics;
* it exposes a slots endpoint unless disabled;
* it has Web UI and agent features that should remain disabled in an untrusted product boundary;
* it supports embedding and reranking modes;
* GGUF is intended to contain the information needed to load a model and is extensible;
* Hugging Face Hub downloads can be pinned to a full commit hash;
* Hugging Face repositories may be gated and require browser acceptance plus an access token for programmatic downloads;
* Hugging Face model cards include publisher-supplied metadata such as licence, base model and intended use;
* Hugging Face scans repository files for malware and pickle imports but does not guarantee safety;
* pickle deserialisation can execute arbitrary code;
* Safetensors is designed for safe tensor storage without pickle execution;
* and ONNX Runtime GenAI's generate API remains documented as preview while offering CPU, CUDA and DirectML packages.

Every runtime commit, server argument, model format version, CUDA package and Windows driver requirement must be revalidated before acceptance and before each production runtime update.

## 11. Terminology

### 11.1 Model Source

A repository, URL, local path, removable medium or curated package from which model artefacts are acquired.

---

### 11.2 Source Revision

An immutable commit, object version, release digest or exact local file snapshot.

---

### 11.3 Model Artefact

One or more immutable files used to represent model weights and required auxiliary data.

---

### 11.4 Model Manifest

An Opure-generated authoritative record describing the verified artefact.

---

### 11.5 Model Identity

The combination of source, source revision, file hashes, architecture, format, quantisation and provenance.

---

### 11.6 Model Record

The persistent Opure registration for one model identity.

---

### 11.7 Derived Model

A model artefact produced by conversion, quantisation, merge or another transformation from a source model.

---

### 11.8 Runtime Package

A verified set of executable and native files implementing one local inference runtime.

---

### 11.9 Runtime Adapter

Trusted Opure code that maps canonical AI Router requests to one runtime package.

---

### 11.10 Execution Profile

A versioned set of runtime, backend, model and resource parameters.

---

### 11.11 Loaded Model Instance

One supervised runtime process with one verified model loaded.

---

### 11.12 Inference Session

A bounded logical request or conversation context managed by the AI Router.

It is not a provider-side persistent conversation.

---

### 11.13 Content-Addressed Store

Storage in which immutable content is located by cryptographic hash.

---

### 11.14 Runtime Cache

Mutable performance data that is not part of model identity.

---

### 11.15 KV Cache

Runtime attention state whose memory use depends on context, model and configuration.

---

### 11.16 Quantisation

A transformation reducing numerical precision and typically memory use.

It can affect model quality and performance.

---

### 11.17 Model Role

The intended runtime operation, such as:

* generation;
* code infill;
* embedding;
* reranking;
* vision-language;
* or audio.

# 12. Options Considered

The principal first-runtime options are:

1. Pinned `llama.cpp` process with GGUF.
2. In-process `llama.cpp` through a .NET binding.
3. ONNX Runtime GenAI.
4. Windows ML or WinML.
5. Ollama-managed runtime.
6. LM Studio local server.
7. LocalAI.
8. Python Transformers runtime.
9. Vendor-specific TensorRT-LLM runtime.
10. Custom native runtime.
11. Remote-only providers.

# 13. Option A — Pinned `llama.cpp` Process

## 13.1 Advantages

* broad local model support;
* GGUF ecosystem;
* strong quantisation support;
* NVIDIA CUDA;
* CPU fallback;
* hybrid offload;
* Windows builds;
* local server;
* embeddings;
* code infill;
* process isolation;
* open source;
* and no Python runtime requirement.

---

## 13.2 Disadvantages

* rapidly changing command and API surface;
* native dependency;
* loopback HTTP;
* server endpoints require hardening;
* GGUF compatibility can change;
* and runtime package maintenance.

---

## 13.3 Mitigations

* pinned commit;
* Opure-built package;
* adapter contract;
* process isolation;
* exact arguments;
* API-key authentication;
* no network;
* endpoint allowlist;
* compatibility suite;
* and versioned execution profiles.

## 13.4 Decision

Selected.

# 14. In-Process Native Binding

Rejected as the first production approach because:

* native crashes could terminate the trusted Runtime;
* GPU resource leaks could destabilise it;
* unload is difficult;
* binding versions add another compatibility layer;
* and same-process native parsing increases the trusted computing base.

A future isolated worker with direct native API may replace loopback HTTP after evidence.

---

# 15. ONNX Runtime GenAI

Attractive because:

* Microsoft maintains it;
* it offers .NET packages;
* it supports CPU, CUDA and DirectML;
* and it provides generation, tokenisation and KV-cache management.

Deferred as the first production runtime because its generate API is currently documented as preview and model preparation remains a separate compatibility burden.

It remains a priority future adapter.

---

# 16. Windows ML

Potentially valuable for Windows-integrated execution and hardware abstraction.

Deferred until its generative-model workflow, model coverage and local coding-model evidence meet Opure requirements.

---

# 17. Ollama

Rejected as the first managed runtime because it introduces:

* another model store;
* another daemon;
* another manifest format;
* its own download and update authority;
* and a broader product lifecycle outside Opure.

Opure may support an explicitly configured unmanaged Ollama endpoint as a local Provider Profile under ADR-0019.

---

# 18. LM Studio

Rejected as the first managed runtime because it is an external desktop product with its own model and server lifecycle.

It may be supported as an unmanaged local endpoint.

---

# 19. LocalAI

Deferred because it adds another server framework and dependency layer.

---

# 20. Python Transformers

Rejected as the Version 1 production runtime because:

* Python environment management;
* native wheel compatibility;
* remote-code risks;
* pickle exposure;
* dependency size;
* and process reproducibility

would substantially increase the attack and support surface.

---

# 21. TensorRT-LLM

Potentially valuable for high NVIDIA performance.

Deferred because it is vendor specific and introduces additional engine-build, compatibility and distribution complexity.

---

# 22. Custom Native Runtime

Rejected because Opure should not build a model inference engine.

---

# 23. Decision

Opure will provisionally adopt:

> **An out-of-process, Opure-built and commit-pinned `llama.cpp` runtime using verified GGUF models, CUDA on the reference NVIDIA workstation, CPU as a separately approved recovery profile, immutable content-addressed model storage, no runtime network or downloads and strict resource, licence, provenance and lifecycle controls.**

This decision is:

* [ ] Permanent until superseded
* [x] Provisional pending runtime, GPU, format and lifecycle evidence
* [ ] Approval of ONNX Runtime GenAI
* [ ] Approval of direct Safetensors execution
* [ ] Approval of automatic conversion or quantisation
* [ ] Approval of multi-GPU or distributed inference

---

One model per process increases load time during switches but provides clear resource release.

# 235. Release Gate

Local model support is blocked when:

* runtime source is not pinned;
* runtime files are mutable;
* runtime package lacks licences or SBOM;
* a model source is mutable without resolved revision;
* a source token appears outside Vault;
* model acquisition can run repository code;
* pickle can be loaded;
* unverified GGUF reaches the runtime;
* model licence is hidden;
* a model can download itself;
* runtime can bind externally;
* runtime accepts unauthenticated requests;
* Web UI or agent features are enabled;
* prompt-cache files persist;
* runtime can access provider credentials;
* runtime can open unexpected network connections;
* runtime silently falls back to CPU or cloud;
* model load can exhaust system resources without preflight;
* child processes can escape supervision;
* output can apply a patch or command directly;
* model revisions can be overwritten;
* deletion can remove shared content unexpectedly;
* or Trust Centre cannot identify the exact runtime and model bytes used.

# 241. Alternatives Rejected

An in-process native binding is rejected initially because native runtime faults and resource leaks should not terminate the trusted Runtime.

ONNX Runtime GenAI is deferred because its generation API is currently preview.

Ollama and LM Studio are not selected as the managed runtime because they introduce independent model and service lifecycles, though they may be configured as unmanaged local Provider Profiles.

Python Transformers is rejected for Version 1 production because environment, pickle, remote-code and dependency risks are disproportionate.

TensorRT-LLM is deferred because vendor-specific engine building and distribution would narrow provider neutrality.

Automatic model download, conversion and quantisation are rejected because each changes licence, bytes, storage and runtime risk.

Runtime cloud fallback is rejected because local failure must not cross the project data boundary.

Persistent provider-style conversations and prompt caches are rejected because Opure owns local context and memory.

# 247. Final Decision Statement

> **Opure will provisionally use a provider-neutral Local Model Runtime boundary whose first production adapter launches an Opure-built, source-pinned `llama.cpp` `llama-server` process with verified GGUF artefacts, CUDA acceleration on the reference NVIDIA workstation and a separately selected CPU recovery profile, while every model is acquired only through an explicit revision-pinned and licence-reviewed plan, quarantined, independently scanned, defensively parsed, recorded in an immutable hash-addressed Model Store, loaded through a resource-budgeted one-model-per-process scheduler, exposed only through authenticated loopback to the AI Router with Web UI, agent, download, RPC and persistent-cache features disabled, and treated as an untrusted content generator whose outputs remain subject to normal review, because local inference must provide real offline control without turning model files, native runtimes, mutable repositories or workstation resources into hidden authority.**
