## 1. Purpose

This specification defines the Trust Centre and security architecture of the Opure Platform.

It establishes how Opure:

- protects developer authority;
- evaluates permissions;
- requests and records approvals;
- classifies data;
- controls secrets;
- mediates network access;
- prevents unauthorised external sharing;
- redacts sensitive information;
- records significant actions;
- isolates and quarantines unsafe components;
- detects abuse or policy violations;
- supports incident investigation;
- retains and deletes security records;
- and recovers safely after failure.

The Trust Centre must make significant Opure activity understandable without exposing secret values or overwhelming the developer with technical noise.

Security controls must remain deterministic.

Artificial intelligence may assist explanation, classification and review.

It must not grant permission, weaken policy or become the authority for security decisions.

---

## 2. Founding Rule

> **Trust must be inspectable, permissions must be explicit, and secrets must never become ordinary data.**

Opure must never rely on hidden trust.

A component is not trusted merely because it:

- is installed;
- is local;
- is signed;
- was generated by AI;
- is built by Opure;
- uses MCP;
- runs on localhost;
- or worked safely before.

Trust must be based on:

- identity;
- capability;
- scope;
- policy;
- approval;
- evidence;
- isolation;
- and current runtime state.

## 4. Security Goals

Opure security must protect:

- project source code;
- project metadata;
- credentials;
- private keys;
- tokens;
- environment secrets;
- local files;
- build systems;
- repositories;
- developer identity;
- provider accounts;
- cloud data;
- workflow authority;
- plugin boundaries;
- MCP boundaries;
- network destinations;
- audit integrity;
- and project isolation.

Security must preserve:

- human control;
- least privilege;
- local-first operation;
- visible external sharing;
- safe recovery;
- and developer data ownership.

---

## 5. Non-Goals

This specification does not claim that Opure can:

- eliminate every vulnerability;
- guarantee external provider behaviour;
- guarantee third-party plugin safety;
- guarantee model output safety;
- prevent all operating-system compromise;
- provide enterprise identity federation in the first release;
- replace full endpoint-security software;
- or prove that generated code is secure.

Opure must communicate limitations honestly.

# 7. Security Principles

## 7.1 Human Authority

The developer is the final authority for high-impact actions.

---

## 7.2 Least Privilege

Services, workflows, plugins, MCP servers and external tools must receive only the permissions required for the current action.

---

## 7.3 Default Deny

Undeclared or unapproved protected actions must be denied.

---

## 7.4 No Ambient Authority

A process does not gain broad authority merely because it runs inside Opure.

---

## 7.5 Explicit External Sharing

Project data must not leave the approved local boundary without policy permission and visible disclosure.

---

## 7.6 Defence in Depth

Security must not depend on one control.

Relevant controls may include:

- policy evaluation;
- process isolation;
- filesystem boundaries;
- network mediation;
- secret detection;
- redaction;
- approval;
- journalling;
- Trust Centre recording;
- and recovery.

---

## 7.7 Fail Securely

When a security-critical service fails, Opure must prefer blocking protected actions over continuing without enforcement.

---

## 7.8 Honest Limits

Opure must not claim visibility or control that a provider or operating system does not expose.

---

# 8. Trust Model

## 8.1 Trusted Core

The trusted core should be kept as small as practical.

Likely trusted-core components include:

- Runtime Bootstrap;
- Service Registry;
- Lifecycle Manager;
- Policy Engine;
- Approval Manager;
- Secrets Vault interface;
- secure configuration loader;
- Audit Integrity Service;
- Trust Centre writer;
- and Workspace transaction coordinator.

The exact trusted-core boundary must be recorded in an ADR.

---

## 8.2 Semi-Trusted Built-In Components

Built-in services outside the trusted core may be considered semi-trusted.

They must still use explicit capabilities and policy checks.

---

## 8.3 Untrusted Components

The following must be treated as untrusted or externally controlled:

- third-party plugins;
- unsigned local plugins;
- MCP servers;
- provider adapters;
- AI providers;
- model output;
- project build scripts;
- package-manager scripts;
- repository hooks;
- CLI tools;
- browser automation;
- downloaded code;
- imported workflow definitions;
- imported knowledge packs;
- and external API responses.

---

## 8.4 Trust Is Contextual

A component may be trusted for one narrow capability and untrusted for another.

Example:

```text
Plugin A:
    trusted to read selected project metadata
    not trusted to access network
    not trusted to read secrets
    not trusted to write files directly
```

---

## 8.5 Trust Expiry

Trust decisions may expire because of:

- time;
- plugin update;
- provider endpoint change;
- workflow change;
- project policy change;
- signing-key revocation;
- or runtime integrity failure.

---

# 9. Protected Actions

The following actions are protected and require deterministic policy evaluation:

- reading project files outside approved scope;
- writing project files;
- applying patches;
- executing commands;
- running project scripts;
- installing dependencies;
- accessing secrets;
- transmitting data externally;
- invoking remote AI providers;
- connecting to MCP servers;
- installing or updating plugins;
- changing security policy;
- elevating privileges;
- deleting project data;
- publishing artefacts;
- modifying repository history;
- and performing irreversible operations.

Additional actions may be classified as protected by policy.

---

# 10. Policy Engine

## 10.1 Responsibility

The Policy Engine evaluates whether a requested protected action is:

- allowed;
- denied;
- allowed with conditions;
- or requires approval.

---

## 10.2 Deterministic Evaluation

Policy evaluation must be deterministic for the same policy version and request facts.

AI must not determine the final result.

---

## 10.3 Policy Inputs

A policy request should include:

- initiating user;
- initiating service;
- plugin or MCP identity where applicable;
- project;
- workflow;
- stage;
- requested capability;
- target;
- data classification;
- destination;
- provider;
- model;
- risk level;
- reversibility;
- active approvals;
- current runtime mode;
- and policy version.

---

## 10.4 Policy Result

A policy result should include:

- decision;
- stable reason code;
- human-readable explanation;
- conditions;
- required approval;
- approval scope;
- expiry;
- and Trust Centre correlation.

---

## 10.5 Decision Types

Recommended decisions are:

- **Allow**
- **Allow with Conditions**
- **Approval Required**
- **Deny**
- **Unavailable Safely**
- **Recovery Required**

---

## 10.6 Conditions

Conditions may include:

- selected files only;
- local provider only;
- approved domain only;
- one-time execution;
- read-only;
- maximum data size;
- no secret content;
- expiry;
- and explicit patch review.

## 12.2 Permission Subjects

Subjects may include:

- user;
- service;
- workflow;
- workflow stage;
- plugin;
- MCP server;
- provider adapter;
- CLI adapter;
- and external connector.

---

## 12.3 Scope

Scope may include:

- one action;
- selected file;
- selected directory;
- file pattern;
- project;
- provider;
- domain;
- secret reference;
- workflow instance;
- session;
- or bounded time period.

---

## 12.4 Permission Inheritance

Permissions must not broaden through inheritance.

A sub-workflow or child task receives only explicitly delegated scope.

---

## 12.5 Revocation

Permissions must be revocable.

Revocation must stop future protected actions.

Active long-running actions should be cancelled where safe and applicable.

---

## 12.6 Permission Storage

Permission records must not contain secret values.

# 13. Approval System

## 13.1 Purpose

The Approval System converts a developer's informed decision into a bounded authorisation record.

---

## 13.2 Approval Request

An approval request should show:

- action;
- initiating component;
- purpose;
- affected project;
- affected files;
- command;
- provider;
- model;
- network destination;
- data categories;
- detected secrets;
- requested secret references;
- reversibility;
- risk;
- and alternatives.

---

## 13.3 Approval Types

Supported approval types should include:

- approve once;
- approve this stage;
- approve this workflow instance;
- approve this project action class;
- approve this provider for bounded scope;
- approve until expiry;
- deny;
- modify scope;
- and cancel operation.

---

## 13.4 Approval Token

An approval token should include:

- token identifier;
- subject;
- action;
- scope;
- project;
- target;
- data class;
- destination;
- expiry;
- single-use state;
- policy version;
- and revocation state.

---

## 13.5 Material Change

Approval becomes invalid when a material fact changes.

Examples include:

- additional files;
- broader directory scope;
- different command;
- different provider;
- different model class where policy matters;
- new network destination;
- more data;
- detected secret;
- changed workflow stage;
- changed plugin version;
- or increased risk.

---

## 13.6 Approval Reuse

Approval reuse must be explicit and bounded.

The default should be one-time approval for high-risk actions.

---

## 13.7 Denial

A denial must be respected.

A workflow or plugin must not repeatedly pressure the developer to approve the same unchanged action.

## 14.1 Risk Dimensions

Risk should consider:

- confidentiality;
- integrity;
- availability;
- external exposure;
- reversibility;
- privilege;
- scope;
- automation level;
- data sensitivity;
- and component trust.

---

## 14.2 Risk Levels

Recommended risk levels are:

- **Low**
- **Moderate**
- **High**
- **Critical**

---

## 14.3 Deterministic Rules

Risk classification must have deterministic baseline rules.

# 15. Data Classification

## 15.1 Purpose

Data classification helps determine:

- who may access data;
- whether it may leave the machine;
- how it is retained;
- how it is redacted;
- and how it is audited.

---

## 15.2 Recommended Classes

### Public

Information deliberately intended for public release.

### Project Internal

Normal source code, documentation and metadata intended to remain within the project boundary.

### Sensitive

Information requiring stronger handling, such as private business logic, customer data, internal endpoints or non-public architecture.

### Secret

Credentials, private keys, tokens, passwords and equivalent authentication material.

### Restricted External Sharing

Data that may be used locally but must not be sent externally without explicit exceptional approval.

---

## 15.3 Classification Sources

Classification may come from:

- built-in rules;
- file patterns;
- project policy;
- developer labels;
- secret detection;
- plugin metadata;
- or workflow context.

---

## 15.4 Classification Conflicts

The more restrictive classification should apply when evidence conflicts.

---

## 15.5 Classification Inheritance

Derived content should inherit the strongest relevant classification from its sources unless safe redaction or abstraction is verified.

---

# 16. Secret Definition

Secrets include, but are not limited to:

- passwords;
- API keys;
- OAuth tokens;
- personal access tokens;
- private keys;
- signing keys;
- session cookies;
- connection strings containing credentials;
- database passwords;
- recovery codes;
- certificate private material;
- cloud credentials;
- and authentication-bearing environment values.

A value may be secret even if it is stored in a normal text file.

---

# 17. Secrets Vault

## 17.1 Responsibility

The Secrets Vault stores and releases secret values through an encrypted, policy-controlled interface.

---

## 17.2 Storage

Secret values must not be stored in:

- ordinary configuration;
- project memory;
- vector indexes;
- logs;
- Trust Centre records;
- workflow checkpoints;
- plugin storage;
- or normal application databases.

---

## 17.3 Encryption

Secrets must be encrypted at rest.

The first Windows implementation should use operating-system-backed protection where appropriate.

---

## 17.4 Secret Metadata

Metadata may include:

- secret identifier;
- display label;
- type;
- owning user;
- project scope;
- provider or destination;
- creation date;
- last use;
- expiry;
- and revocation state.

Metadata must not reveal the secret value.

---

## 17.5 Secret Access

Secret access must be:

- policy checked;
- scoped;
- time bounded where practical;
- destination bound where appropriate;
- auditable;
- and revocable.

---

## 17.6 Secret Release

Where possible, the Vault should avoid returning raw secret values to callers.

It may instead:

- inject credentials into a controlled request;
- provide a short-lived handle;
- set a restricted process environment;
- or sign a request.

---

## 17.7 Secret Display

The interface must not reveal full secret values by default.

A deliberate reveal action should require stronger confirmation.

---

## 17.8 Secret Deletion

Deletion must remove active secret material and revoke future use.

Backup behaviour must be explained honestly.

---

# 18. Secret Detection

## 18.1 Detection Points

Secret detection must run before:

- external AI requests;
- external API requests;
- network uploads;
- patch application;
- Git staging or commit where configured;
- memory ingestion;
- embedding generation;
- Trust Centre storage;
- diagnostic export;
- and plugin data export.

---

## 18.2 Detection Methods

Detection may use:

- known token patterns;
- entropy analysis;
- key-file recognition;
- environment-file rules;
- provider-specific formats;
- structured parsers;
- and developer-defined patterns.

---

## 18.3 False Positives

Detection results must support developer review.

A developer may mark a false positive.

That decision should be scoped and recorded.

---

## 18.4 False Negatives

Opure must not claim secret detection is perfect.

---

## 18.5 Sensitive File Defaults

The following should be protected by default:

- `.env`;
- private-key files;
- credential stores;
- cloud configuration containing credentials;
- password exports;
- browser credential files;
- and known token caches.

---

# 19. Redaction Service

## 19.1 Purpose

The Redaction Service removes or masks sensitive content before storage, display or external sharing.

---

## 19.2 Redaction Types

Redaction may include:

- full removal;
- partial masking;
- token replacement;
- path minimisation;
- identifier hashing;
- content summarisation;
- and metadata-only representation.

---

## 19.3 Redaction Provenance

A redacted artefact should identify:

- source;
- redaction method;
- rules used;
- date;
- and whether review occurred.

---

## 19.4 Reversibility

Redaction output must not imply reversibility unless original content is retained securely.

---

## 19.5 Redaction Validation

High-risk exports should validate that secret patterns no longer remain.

---

Projects should support:

- **Local Only**

Under Local Only:

- remote embeddings are prohibited;
- remote reranking is prohibited;
- cloud plugin requests are blocked unless separately authorised outside project data scope;

Every external project-data request requires explicit approval.

---

## 21.4 Approved Providers Only

Only approved provider identities and scopes may receive project data.

---

## 21.5 Custom

Custom policy may define:

- provider allow-list;
- domain allow-list;
- data-type restrictions;
- file exclusions;
- maximum size;

- retention requirements;

# 24. Audit Integrity

## 24.1 Integrity Goal

Trust records should be tamper-evident where practical.

---

## 24.2 Append-Oriented Storage

Significant audit history should use append-oriented storage semantics.

Corrections should create new records rather than silently rewriting history.

---

## 24.3 Integrity Techniques

Techniques may include:

- chained hashes;
- signed checkpoints;
- immutable segments;
- monotonic sequence numbers;
- and secure timestamps.

Exact technology is DEFERRED.

---

## 24.4 Clock Integrity

Records should include a monotonic or sequence-based ordering mechanism in addition to wall-clock time where useful.

---

## 24.5 Audit Failure

If durable audit recording is unavailable:

- low-risk read-only activity may continue under policy;
- high-risk protected actions should be blocked or require explicit degraded-mode approval;
- and buffered events must be preserved where possible.

---

# 25. Audit Privacy

## 25.1 No Secret Values

Trust records must never contain:

- passwords;
- private keys;
- raw tokens;
- recovery codes;
- or secret-bearing request headers.

---

## 25.2 Content Minimisation

Full project content should not be duplicated into audit storage unless required and approved.

References, hashes and safe summaries should be preferred.

---

## 25.3 Personal Data

Audit records containing personal data must follow retention and access policy.

---

## 25.4 Export Review

The developer must inspect audit exports before external sharing.

---

# 28. Quarantine

## 28.1 Purpose

Quarantine disables or isolates a component that may be unsafe.

---

## 28.2 Quarantine Targets

Targets may include:

- plugin;
- MCP server;
- provider adapter;
- workflow definition;
- imported knowledge pack;
- external connector;
- or downloaded package.

---

## 28.3 Quarantine Triggers

Triggers may include:

- integrity failure;
- invalid signature;
- repeated crashes;
- undeclared capability use;
- denied filesystem access attempt;
- denied network attempt;
- secret-handling violation;
- malicious payload;
- or operator action.

---

## 28.4 Quarantine Behaviour

Quarantine should:

- stop active execution;
- revoke temporary permissions;
- disconnect network access;
- preserve evidence;
- prevent automatic restart;
- and mark dependent capabilities unavailable.

---

## 28.5 Restoration

Restoring from quarantine requires:

- cause review;
- package or configuration validation;
- policy evaluation;
- and explicit approval.

---

# 29. Safe Mode

## 29.1 Purpose

Safe Mode starts Opure with restricted optional components.

---

## 29.2 Restrictions

Safe Mode should disable or restrict:

- third-party plugins;
- non-essential MCP servers;
- custom provider adapters;
- automatic workflows;
- background network activity;
- and untrusted imported packages.

---

## 29.3 Availability

Core project inspection, policy, Trust Centre and recovery capabilities should remain available where possible.

---

# 30. Incident Management

## 30.1 Security Incident

A security incident is an event that may have caused or attempted:

- secret exposure;
- unauthorised access;
- unauthorised external sharing;
- project corruption;
- audit tampering;
- privilege escalation;
- or unsafe code execution.

---

## 30.2 Incident Record

An incident record should include:

- incident identifier;
- severity;
- discovery time;
- affected project;
- affected components;
- evidence;
- actions taken;
- quarantines;
- possible data exposure;
- recovery state;
- and closure notes.

---

## 30.3 Incident Workflow

A security incident workflow should support:

1. contain;
2. preserve evidence;
3. stop unsafe activity;
4. identify scope;
5. inspect approvals and permissions;
6. revoke access;
7. rotate affected secrets where needed;
8. repair;
9. validate;
10. and document outcome.

---

## 30.4 AI Assistance

AI may help summarise incident evidence.

It must not decide final containment or secret rotation policy.

---

# 31. Recovery

## 31.1 Security Recovery

After a security-related failure, Opure should:

- enter Safe or Recovery Mode;
- preserve audit evidence;
- validate policy integrity;
- validate secret storage;
- inspect incomplete actions;
- inspect quarantined components;
- revoke temporary approvals;
- and require explicit restart of unsafe components.

---

## 31.2 Approval Invalidation

Abnormal shutdown should invalidate temporary approvals unless policy explicitly preserves them safely.

---

## 31.3 Secret Recovery

Secret corruption or Vault failure must block dependent actions.

---

## 31.4 Audit Recovery

Audit journals must support detection of incomplete record writes.

---

# 32. Path and File Security

## 32.1 Workspace Boundary

All project file access must remain within approved workspace boundaries.

---

## 32.2 Path Traversal

Path traversal must be rejected.

# 53. Initial Implementation Milestone

The first Trust Centre and Security milestone is successful when Opure can:

1. load secure default policy;
2. evaluate a protected action deterministically;
3. return Allow, Approval Required or Deny;
4. create a bounded one-time approval;
5. invalidate approval after material change;
6. classify project data;
7. detect common secrets;
8. prevent secrets entering logs;
9. store credentials in an encrypted Vault;
10. authorise secret use without ordinary storage;
11. mediate a remote request through Network Gateway;
12. block remote project-data sharing under Local Only;
13. show an external data-sharing preview;
14. record significant activity in the Trust Centre;
15. correlate workflow, AI, patch and network activity;
16. verify basic audit integrity;
17. quarantine a misbehaving plugin;
18. enter Safe Mode;
19. produce a redacted diagnostic bundle;
20. and recover after abnormal shutdown without reusing unsafe temporary approval.

# 54. Acceptance Criteria

SPEC-008 is implemented when:

- [ ] Protected actions always invoke deterministic policy evaluation.
- [ ] AI cannot grant permission or reduce risk.
- [ ] Undeclared protected actions are denied.
- [ ] Permissions are explicit, scoped, revocable and inspectable.
- [ ] Approval is bound to the exact or bounded action shown.
- [ ] Material changes invalidate approval.
- [ ] High-risk approvals default to one-time use.
- [ ] Project cloud modes are enforced.
- [ ] Local Only prevents remote project-data transmission.
- [ ] External sharing shows destination, data and purpose.
- [ ] Data classification propagates to derived content.
- [ ] Secret values remain outside normal storage.
- [ ] The Vault encrypts secrets at rest.
- [ ] Secret access is scoped and auditable without recording values.
- [ ] Secret detection runs before external sharing, patches, Git, memory and diagnostics.
- [ ] Sensitive files are protected by default.
- [ ] Redaction is traceable and validated.
- [ ] Protected network traffic uses the Network Gateway.
- [ ] Redirects and endpoint changes trigger re-evaluation where relevant.
- [ ] Localhost is not treated as automatic trust.
- [ ] Trust Centre records correlate significant activity.
- [ ] Trust records contain no secret values.
- [ ] Audit integrity is verifiable.
- [ ] Audit failure blocks or constrains high-risk actions.
- [ ] Third-party plugins can be quarantined.
- [ ] MCP servers remain external trust boundaries.
- [ ] Model output remains untrusted.
- [ ] Build scripts and package scripts are treated as executable risk.
- [ ] Privilege elevation is exceptional and recorded.
- [ ] Safe Mode disables untrusted optional components.
- [ ] Security incidents can be contained and investigated.
- [ ] Temporary approvals are invalidated after unsafe restart.
- [ ] Diagnostic exports are redacted and reviewable.
- [ ] External telemetry is disabled by default.
- [ ] Retention and deletion are visible and honest.
- [ ] Security tests include adversarial cases.
- [ ] The platform remains usable offline without cloud security services.

# 58. Founder Approval

This document remains a founder draft until explicitly approved.

Approval establishes the following rules:

- Trust must be inspectable.
- Permissions must be explicit.
- Security decisions are deterministic.
- AI cannot grant authority.
- Secrets never become ordinary project, memory, log or audit data.
- External project-data sharing is visible and policy controlled.
- Local Only means no remote project-data transmission.
- Approval applies only to the action actually shown.
- Material changes require renewed approval.
- Localhost, signatures and installation do not create automatic trust.
- Plugins, MCP servers, models and external tools remain bounded trust domains.
- Significant actions are correlated in the Trust Centre.
- Audit history is privacy-preserving and tamper-evident where practical.
- Unsafe components can be quarantined.
- Safe Mode and Recovery Mode preserve developer control after failure.
- Telemetry is disabled by default.
- Security exists to protect developer freedom, not to conceal platform behaviour.
